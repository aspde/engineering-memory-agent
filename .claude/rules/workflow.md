# 工作流规则

## 适用范围

以下流程适用于：

- 新增功能模块
- 修改公共接口或数据模型
- 架构调整
- 新增依赖

以下情况可直接执行，无需方案确认：

- 修复 bug（范围明确的）
- 添加或修改测试
- 文档修正（typo、格式、措辞）
- 代码格式调整
- 模块内部重构（不改变公共接口）

## 开发流程

需要确认的操作遵循：

分析 → 方案设计 → 用户确认 → 实施 → 验证

### 修改前要求

1. 理解当前项目结构和相关代码
2. 分析需求影响范围
3. 输出实施方案
4. 等待用户确认

收到明确确认后方可：修改文件、创建文件、删除文件、安装依赖、修改配置。

### 禁止

- 未分析代码直接修改
- 根据猜测实现功能
- 重复创建已有能力

## 异步优先

涉及 IO 操作时优先使用 async/await：

- LLM API 调用
- HTTP 请求
- 数据库访问
- 向量检索
- 文件 IO
- 外部服务调用

优先使用异步客户端和异步数据库驱动。避免在异步流程中执行阻塞操作。如果同步实现更简单且不影响性能，可以使用同步方式。

## 依赖管理

新增依赖前必须说明：

1. 引入原因
2. 解决的问题
3. 是否已有依赖可以实现
4. 对项目复杂度的影响

禁止随意增加依赖、引入重复功能库、自动升级依赖。

## 代码修改原则

- 优先小范围修改、复用已有代码、保持接口稳定、遵循现有代码风格
- 避免大规模重构、无需求优化、提前抽象、修改无关代码

## 文档同步

`docs/` 只记录代码无法表达的信息：决策原因、设计权衡、系统边界。

当修改以下内容时，检查相关 docs：

- 架构变化 → `docs/architecture.md`
- Agent 流程变化 → `docs/agent-design.md`
- Memory 设计变化 → `docs/memory-system.md`
- 部署配置变化 → `docs/deployment.md`

禁止修改架构但不更新文档，禁止文档描述与代码长期不一致。
