# 文档维护规则

## 文档目标

项目文档用于：

- 记录系统设计
- 沉淀技术决策
- 保存研发知识
- 支撑 Memory System 检索


文档应保持：

- 准确
- 简洁
- 与代码一致


---

# 文档目录规范

项目文档统一存放：

```
docs/
```


推荐结构：

```
docs/
├── architecture.md
├── agent-design.md
├── memory-system.md
├── rag-design.md
├── tech-stack.md
├── git-knowledge.md
├── deployment.md
└── decisions/
```


---

# 文档更新规则

以下情况必须同步更新文档：

- 架构变化
- 技术选型变化
- 数据模型变化
- API 变化
- Agent Workflow 变化
- Memory 设计变化


禁止：

- 修改架构但不更新文档
- 文档描述与代码长期不一致


---

# 架构文档规范

架构相关修改需要记录：

包括：

- 当前问题
- 设计方案
- 技术选择
- 影响范围


架构文档应说明：

- 模块职责
- 数据流
- 核心设计原则


---

# ADR 规范

重要技术决策使用 ADR 记录。


ADR 包含：

- 背景
- 问题
- 方案选择
- 最终决定
- 优缺点


示例：

```
docs/decisions/
├── ADR-001-tech-stack.md
├── ADR-002-vector-storage.md
```


---

# Memory 文档要求

Memory 相关设计必须记录：

- Memory 类型
- 数据结构
- 存储方式
- 检索流程
- 生命周期管理


---

# 文档修改原则

修改文档时：

优先：

- 保持结构清晰
- 使用准确描述
- 更新相关引用


避免：

- 编写与实现无关的内容
- 添加未经验证的设计
- 重复已有文档

- 技术方案

相关文档是否需要同步更新。
