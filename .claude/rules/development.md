# 开发规则

## 开发流程

所有开发任务必须遵循：

分析 → 方案设计 → 用户确认 → 实施 → 验证


---

## 修改前必须规划

在执行任何会改变项目状态的操作之前，必须先输出实施方案。


包括但不限于：

- 修改代码
- 创建文件
- 删除文件
- 移动文件
- 修改项目结构
- 修改配置文件
- 修改数据库结构
- 增加依赖


---

## 修改前分析要求

执行修改前必须：

1. 理解当前项目结构

2. 查看相关代码实现

3. 分析需求影响范围

4. 提供实施方案

5. 等待用户确认


禁止：

- 未分析代码直接修改
- 根据猜测实现功能
- 重复创建已有能力


---

## 用户确认机制

方案输出后，必须等待用户明确确认，收到确认后方可执行。


未收到明确确认时，禁止：

- 修改文件
- 创建文件
- 删除文件
- 安装依赖
- 修改配置


---

## 异步优先原则

涉及 IO 操作时，优先使用异步实现。


优先异步处理：

- LLM API 调用
- HTTP 请求
- 数据库访问
- 向量检索
- 文件 IO
- 外部服务调用


优先使用：

- async / await
- 异步客户端
- 异步数据库驱动


避免：

- 在异步流程中执行阻塞操作
- 使用同步调用阻塞事件循环


如果同步实现更简单且不会影响性能，可以使用同步方式。


---

## 依赖管理

新增依赖前必须说明：

1. 引入原因

2. 解决的问题

3. 是否已有依赖可以实现

4. 对项目复杂度的影响


禁止：

- 随意增加依赖
- 引入重复功能库
- 自动升级依赖


---

## 代码修改原则

修改代码时：

优先：

- 小范围修改
- 复用已有代码
- 保持接口稳定
- 遵循现有代码风格


避免：

- 大规模重构
- 无需求优化
- 提前抽象
- 修改无关代码


---

## 简单优先原则

实现功能时：

优先选择：

- 简单方案
- 可维护方案
- 易理解方案


避免：

- 过度设计
- 提前优化
- 为未来需求增加复杂度
