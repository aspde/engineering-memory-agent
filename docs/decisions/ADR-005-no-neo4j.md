# ADR-005: 拒绝引入 Neo4j，坚持 PostgreSQL 单存储

**日期**: 2026-07-20

**状态**: 已接受

## 背景

在引入三阶段抽取（摘要→实体→关系）和图向量混合检索时，提出是否引入 Neo4j 作为图数据库来存储实体关系。

## 决策

**不引入 Neo4j**，继续使用 PostgreSQL + pgvector 作为唯一数据存储方案。

## 理由

1. **检索模式不匹配**：EMA 的核心场景是语义搜索（"我们之前怎么处理 X 问题的？"），不是图遍历——用户不会沿着实体关系图跳转查询。图数据库的优势在这个场景中不是主要检索路径。
2. **pgvector 足够**：向量召回 + 全文检索 + 时间过滤覆盖了 EMA 的实际需求。实体关系可以以 JSON 字段存于 PG 行中，随向量召回的记录一同返回，无需图遍历来发现。
3. **避免过度设计**：目前对图查询的具体需求是推测性的。引入 Neo4j 增加部署复杂度、双写一致性负担，而换不来可验证的用户价值。
4. **约束一致性**：保持 `constraints.md` 中"PostgreSQL + pgvector 作为唯一数据存储方案"的约束不变。

## 后果

- 图扩展通过 PostgreSQL 递归 CTE 实现，N > 3 跳时性能可能下降
- 未来若出现真正需要复杂图遍历的场景，可重新评估（届时会有真实数据支撑）
- 坚持简单优先原则：先做好单存储，场景验证后再说
